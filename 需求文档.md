# Polymarket 自动跟单机器人（Copy Trading）需求文档（含技术方案）

> 后续开发以本文件为准。

## 0. 范围与结论（先读这个）
- **后端**：`dapp-api/`（Laravel 10.10 + Sanctum + dcat-plus/laravel-admin）。
- **前端**：`dapp-h5/`（目前为空目录，按 **Vue3 + Vant** 从 0 初始化）。
- **跟单授权方式**：**托管机器人私钥**（后端全自动下单）。
- **触发源**：**按 leader 成交复制**（不是复刻挂单/撤单）。
- **金额单位**：跟单比例与 min/max 均按 **USDC 金额**。
- **技术关键点**：Polymarket CLOB 下单需要 **EIP-712 订单签名** + **L2 HMAC headers（POLY_*）**。现已确定采用 **纯 PHP 实现**：
  - 使用 `polymarket-php/polymarket` 负责 CLOB/Gamma/Data 的请求封装与 L1/L2 鉴权
  - 使用 `SleepFinance\\Eip712` 生成 EIP-712 typed data digest
  - 使用 `kornrunner/secp256k1` 对 digest 做 secp256k1 签名，生成 `order.signature`
  - 由 Laravel 直接完成签名与下单（不再引入 Node `dapp-worker`）

## 1. 背景
开发一个 Polymarket 自动跟单系统，用户登录后可查看资产与收益，并创建跟单任务，系统根据 leader 的成交自动复制下单。

Polymarket 官方 API 文档：
- https://docs.polymarket.com/api-reference/introduction

## 2. 术语
- **Leader**：被跟单的地址/账户。
- **Follower**：创建跟单任务的用户。

## 4. 非功能需求（必须明确）

### 4.1 安全
- 托管私钥必须加密存储；私钥明文不得写日志、不得落盘。
- 登录 nonce 必须一次性、短 TTL、不可重放。
- 纯 PHP 实现：不再引入 Node 内部服务；敏感操作集中在 Laravel 后端进程内完成（仍需保证私钥不落盘不写日志）。

### 4.2 稳定性与幂等
- leader 成交去重（trade_id 唯一）
- order_intent 去重（同一任务 + 同一成交只能生成一次）
- 下单失败要有错误码记录与重试策略（网络可重试，余额不足不可重试）

### 4.3 计算精度
- 金额统一用 **整数最小单位**存储：USDC = 1e6。

## 6. 总体架构（纯 PHP 实现）

### 6.1 为什么不再需要 Node（dapp-worker）
Polymarket CLOB 下单同时需要：
- HTTP 层：L2 HMAC 认证 headers（`POLY_*`）
- 订单本体：EIP-712 签名字段 `order.signature`

目前你已安装：
- `polymarket-php/polymarket`：提供 CLOB/Gamma/Data 的请求封装，并实现 CLOB L1/L2 鉴权（含 HMAC signer）
- `SleepFinance\\Eip712`：可生成 `eth_signTypedData_v4` 规则的 typed data digest
- `kornrunner/secp256k1`（已在 vendor 中）：可对 digest 进行 secp256k1 签名，得到 r/s/v 与拼接 signature

因此我们采用 **纯 PHP**：由 Laravel 直接完成「订单 typed data 构建 → digest → 签名 → 调用 CLOB 下单」的全链路。

### 6.2 模块划分
- **Laravel（dapp-api）**：
  - 登录/用户（nonce + personal_sign）
  - 托管钱包/凭证管理（私钥加密存储）
  - leader/copy task 管理
  - 成交轮询、意图生成、风控与落库
  - EIP-712 订单签名与 CLOB 下单（新增 `PolymarketOrderSigner` / `PolymarketOrderService`）
  - 首页/收益/记录/社区 API
  - 后台 dcat 管理

## 11. 纯 PHP 下单与签名模块（替代 Node dapp-worker）

### 11.1 目标
在 Laravel 内部实现：
- 构建 Polymarket CLOB 下单所需的 **Signed Order**（包含 `order.signature`）
- 使用 `polymarket-php/polymarket` 完成 CLOB 的 L1/L2 鉴权与下单请求

### 11.2 依赖
- `polymarket-php/polymarket`（已安装）
- `SleepFinance\\Eip712`（已接入 autoload，路径见 `composer.json`）
- `kornrunner/secp256k1`（vendor 已存在）

### 11.3 需要新增的核心类（后端）
- `App\\Services\\Polymarket\\OrderTypedDataBuilder`
  - 输入：订单参数（maker/signer/funder、token_id、side、price、size、salt、expiration、feeRateBps…）
  - 输出：EIP-712 typed data（types/domain/primaryType/message）

- `App\\Services\\Polymarket\\OrderSigner`
  - 调 `SleepFinance\\Eip712->hashTypedDataV4()` 得到 digest
  - 用 `kornrunner\\Secp256k1` 对 digest 签名，得到 r/s/v
  - 输出：`0x{r}{s}{v}` 格式的 `order.signature`（并填充 `signatureType`）

- `App\\Services\\Polymarket\\ClobTradingService`
  - 封装：创建/派生 API key、生成 L2 headers、POST /order、GET /balance-allowance、GET /book 等
  - 内部使用 `PolymarketPhp\\Polymarket\\Clob` 与 `ClobAuthenticator`

### 11.4 校验（PoC 验收点）
- 给定一个测试私钥 + token_id + 价格/数量：能成功生成 `order.signature`
- 调用 `POST /order` 返回订单 id（或明确的错误码），证明签名与 headers 均正确

## 14. 跟单逻辑与任务参数说明（运维/配置参考）

### 14.1 当前系统的真实跟单逻辑
当前系统采用“**按 leader 成交复制**”模式，不是复制挂单，也不是直接按张数复制，而是：

1. 后台轮询 leader 最近成交
   - 入口：[dapp-api/app/Console/Commands/PmPollLeaderTradesCommand.php](dapp-api/app/Console/Commands/PmPollLeaderTradesCommand.php)
   - 行为：拉取 leader 最近 trades，写入 `pm_leader_trades`

2. 根据启用中的 copy task 生成跟单意图
   - 入口：[dapp-api/app/Jobs/PmCreateOrderIntentsJob.php](dapp-api/app/Jobs/PmCreateOrderIntentsJob.php)
   - 逻辑：
     - `target_usdc = leader_trade.size_usdc * ratio_bps / 10000`
     - 再按 `min_usdc / max_usdc` 做 clamp
     - 生成 `pm_order_intents`

3. 执行跟单意图并尝试真实下单
   - 入口：[dapp-api/app/Jobs/PmExecuteOrderIntentJob.php](dapp-api/app/Jobs/PmExecuteOrderIntentJob.php)
   - 执行前会校验：
     - 是否有托管钱包
     - 是否有 token_id
     - token 是否可交易
     - 价格是否合法
     - side 是否为 BUY/SELL
     - clamped_usdc 是否大于 0
     - 计算出的 size 是否大于 0
   - 全部通过后才会调用 [dapp-api/app/Services/Pm/PolymarketTradingService.php](dapp-api/app/Services/Pm/PolymarketTradingService.php) 下单

4. 下单结果写入订单表
   - 表：`pm_orders`
   - 首页最近记录、我的记录，本质上都以订单表为基础展示

### 14.2 任务参数含义

#### 1) `leader_id`
表示“跟谁”。
- 绑定被跟单对象（leader）
- 只有这个 leader 的成交会触发当前任务

#### 2) `status`
表示任务是否启用。
- `1`：启用
- 非 `1`：暂停/关闭
- 只有启用状态才会参与生成跟单意图

#### 3) `ratio_bps`
表示跟单比例，单位是 **bps（万分比）**。
- `10000 = 100%`
- `5000 = 50%`
- `2000 = 20%`
- `1000 = 10%`
- `500 = 5%`

示例：
- leader 本次成交金额是 `100 USDC`
- 如果 `ratio_bps = 1000`
- 则目标跟单金额 = `10 USDC`

#### 4) `min_usdc`
表示**单笔最低跟单金额**。
- 如果按比例算出来太小，就抬到这个值
- 作用是避免太小金额导致最终 `size` 过小、无法有效下单

#### 5) `max_usdc`
表示**单笔最高跟单金额**。
- 如果按比例算出来太大，就压到这个值
- 作用是限制单笔风险，避免被 leader 大单带过头

#### 6) `max_slippage_bps`
表示**最大允许滑点**。
- 当前代码里已作为风控快照保存
- 但执行层尚未完整启用
- 当前版本可视为“预留参数”

#### 7) `allow_partial_fill`
表示**是否允许部分成交**。
- 当前代码里已入库，但执行层尚未完整启用
- 当前版本可视为“预留参数”

#### 8) `daily_max_usdc`
表示**每日最大跟单总金额**。
- 当前代码里已保留，但尚未完整执行拦截
- 当前版本可视为“预留参数”

### 14.4 一笔跟单是怎么算出来的
示例：
- leader 成交：`BUY @ 0.40`
- leader 成交金额：`100 USDC`
- 任务参数：
  - `ratio_bps = 1000`（10%）
  - `min_usdc = 5 USDC`
  - `max_usdc = 20 USDC`

计算过程：
1. `target_usdc = 100 * 10% = 10 USDC`
2. 套用 min/max 后，`clamped_usdc = 10 USDC`
3. 用 leader 成交价换算本次下单数量：
   - `size = 10 / 0.40 = 25 shares`
4. 最终按：
   - side = BUY
   - price = 0.40
   - size = 25
   去尝试下单

### 14.5 金额填写单位说明
系统内部金额统一使用 **USDC 最小单位（1e6）**。

换算示例：
- `1 USDC = 1000000`
- `2 USDC = 2000000`
- `5 USDC = 5000000`
- `10 USDC = 10000000`
- `20 USDC = 20000000`

### 14.8 常见踩坑

#### 1) `min_usdc` 设得太小
例如：
- `min_usdc = 10000`，其实只等于 `0.01 USDC`
- 很容易导致最终 `size` 过小，落成 `invalid_size`

建议：
- 最低不要低于 `1000000`（1 USDC）

#### 2) `max_usdc` 设得太大
如果总资产不大，但单笔上限过大，会导致：
- 余额不够
- 风险过大
- 更容易连续失败

建议：
- 单笔上限先控制在总资产的 `10% ~ 20%`

#### 3) `ratio_bps` 设得太高
例如：
- `ratio_bps = 8000`，等于 80%
- 本质上接近重仓复制 leader

建议：
- 刚开始不要超过 `1000 ~ 2000`（10% ~ 20%）

### 14.9 当前最推荐的默认配置
如果当前目标是“先稳定跑起来，再看真实成交效果”，建议默认使用：
- `ratio_bps = 500`
- `min_usdc = 1000000`
- `max_usdc = 5000000`

即：
- 跟单比例 5%
- 单笔最少 1 USDC
- 单笔最多 5 USDC

这套参数更适合当前阶段做链路验证与小额实盘。
